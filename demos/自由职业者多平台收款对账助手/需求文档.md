# 自由职业者多平台收款对账助手 - 用户需求文档

> 文档版本：v1.0
> 创建日期：2026-06-28
> 文档状态：初稿
> 适用产品：自由职业者多平台收款对账助手（以下简称"对账助手"）

---

## 1. 文档概述

### 1.1 产品愿景

为自由职业者提供一款轻量级的多平台收款对账工具，聚焦"应收-实收"差异比对这一细分场景，帮助用户高效完成月度对账、差异追踪与报税准备，不再为跨平台账单核对而焦虑。

### 1.2 文档目的

本文档定义"对账助手"产品的用户需求，作为后续产品设计、UI 原型、开发与测试的核心依据。文档聚焦于 **需求层面**，不涉及具体技术实现方案。

### 1.3 适用范围

本文档适用于产品、设计、开发、测试团队，以及参与 MVP 评审的相关方。

### 1.4 术语表

| 术语 | 定义 |
|------|------|
| 应收款 | 用户基于合同、订单或发票，应向客户收取的金额 |
| 实收款 | 用户在各收款平台实际到账的金额（可能扣除手续费、平台佣金等） |
| 对账 | 将应收款与实收款逐笔匹配，识别差异的过程 |
| 差异 | 应收金额与实际到账金额之间的差额，包括未到账、超额、手续费扣除、汇率差等 |
| CSV | 逗号分隔值文件，各平台账单导出的通用格式 |
| 匹配引擎 | 根据金额、时间、备注等字段自动关联应收与实收记录的算法模块 |
| 对账报告 | 按月/季度/年生成的对账结果汇总文档 |

## 2. 目标用户与使用场景

### 2.1 目标用户画像

#### 画像 A：独立设计师
- **典型特征**：30 岁左右，同时承接品牌设计、UI 设计等外包项目
- **收款渠道**：微信收款、支付宝、站酷/花瓣等设计平台结算
- **痛点**：每月需花 2-3 小时核对多平台账单，曾发生过客户声称已付款但实际未到账的情况
- **核心诉求**：快速定位未到账订单，按客户维度生成对账单

#### 画像 B：自由撰稿人
- **典型特征**：向多家媒体、公众号长期供稿，稿费按篇/按月结算
- **收款渠道**：微信、支付宝、稿定/知乎等创作平台
- **痛点**：不同平台的结算周期不一致（月结、季结），难以统一管理
- **核心诉求**：按平台/媒体维度查看待收款项，自动识别逾期未到账

#### 画像 C：独立开发者
- **典型特征**：开发 SaaS 工具或 App，通过订阅、一次性购买等方式变现
- **收款渠道**：Stripe（海外）、支付宝、微信支付、App Store/Google Play
- **痛点**：涉及多币种（CNY/USD/EUR），汇率换算和跨境手续费让核对变得复杂
- **核心诉求**：自动处理汇率换算，区分"平台手续费"与"实际收入"

#### 画像 D：独立咨询师
- **典型特征**：为企业提供管理、IT、财务等咨询服务，按项目/小时收费
- **收款渠道**：银行转账（个人/对公）、微信、支付宝
- **痛点**：需要按项目维度统计收入，为报税准备开票清单
- **核心诉求**：按项目维度生成收入汇总，导出报税所需的收入明细表

### 2.2 典型使用场景

#### 场景 1：月度对账
> 月末，设计师小李打开对账助手，导入本月从微信、支付宝导出的 CSV 账单，系统自动匹配她登记的应收款项。她看到 3 笔差异——2 笔是平台手续费扣除，1 笔是客户声称已付但实际未到账。她将未到账的 1 笔导出为催款提醒，发送给客户。
> **用时：15 分钟（替代过去 2-3 小时的手动核对）。**

#### 场景 2：多币种对账
> 独立开发者老王同时通过 Stripe（USD）和支付宝（CNY）收款。他导入两份 CSV 后，系统按当日汇率自动换算，并标记出汇率波动超过 1% 的差异条目，让他清楚知道哪些差异是汇率造成的、哪些是真正的问题。

#### 场景 3：报税准备
> 咨询师张老师年底需要报税，她通过对账助手导出全年收入汇总表和开票清单，直接交给会计，省去了从多个平台逐一整理账单的麻烦。

#### 场景 4：差异追溯
> 撰稿人小赵在月度对账报告中发现某平台上月有 5 笔"手续费异常"——原本 2% 的费率变成了 3.5%。他导出差异报告，向平台发起申诉并成功追回差额。

## 3. 需求概述

### 3.1 产品定位

**不做通用记账**，而是聚焦于自由职业者"应收-实收"差异比对的细分场景。核心差异化：
- **轻量**：无需手动录入每一笔流水，CSV 导入即可
- **精准**：自动匹配应收与实收，差异一目了然
- **专业**：支持多币种、多平台、多项目维度

## 4. 功能需求

### 4.1 多渠道 CSV 导入

#### 4.1.1 需求描述

用户能够从微信收款、支付宝、Stripe、PayPal、银行等渠道导出 CSV 账单文件，并导入系统。系统自动识别文件格式并解析。

#### 4.1.2 功能要点

| 编号 | 功能点 | 描述 |
|------|--------|------|
| F-1.1 | 预置模板 | 系统预置微信、支付宝、Stripe、PayPal 四大渠道的标准解析模板，用户导入时自动匹配 |
| F-1.2 | 自定义模板 | 用户可为其他渠道创建自定义解析模板，指定字段映射关系 |
| F-1.3 | 字段映射 | 导入时展示字段映射预览界面，用户可调整"交易时间""金额""交易对象""备注"等关键字段的对应关系 |
| F-1.4 | 数据预览 | 导入前展示前 10 条数据的解析预览，用户确认无误后再正式导入 |
| F-1.5 | 重复检测 | 导入时自动检测重复记录（基于交易 ID + 金额 + 时间），防止重复导入 |
| F-1.6 | 批量导入 | 支持一次导入多个 CSV 文件，系统按文件名或内容自动分配渠道 |
| F-1.7 | 导入日志 | 记录每次导入的详情（成功条数、失败条数、失败原因），便于用户追溯 |

#### 4.1.3 验收标准

- 微信/支付宝/Stripe/PayPal 四大渠道的 CSV 导入成功率 ≥ 98%
- 导入预览响应时间 ≤ 2 秒（100 条数据以内）
- 重复记录检测准确率 100%

---

### 4.2 应收账单管理

#### 4.2.1 需求描述

用户需要在系统中登记应收款项（基于合同、订单、发票等），作为对账的基准。

#### 4.2.2 功能要点

| 编号 | 功能点 | 描述 |
|------|--------|------|
| F-2.1 | 手动录入 | 用户可逐笔录入应收款：客户名称、金额、币种、预期到账日期、关联项目、备注 |
| F-2.2 | 批量导入 | 用户可通过 CSV/Excel 模板批量导入应收款记录 |
| F-2.3 | 状态管理 | 应收款状态流转：待收款 → 部分到账 → 已到账 → 已逾期 → 已核销 |
| F-2.4 | 项目关联 | 支持将应收款关联到具体项目，便于按项目维度统计 |
| F-2.5 | 客户管理 | 维护客户信息（名称、联系方式、历史交易记录），便于查询和催款 |
| F-2.6 | 逾期提醒 | 超过预期到账日期未到账的应收款，系统自动标记为"逾期"并提醒用户 |

#### 4.2.3 验收标准

- 支持至少 3 种币种（CNY、USD、EUR）
- 批量导入支持单次 1000 条以上
- 逾期提醒在到期日当天自动触发

### 4.3 自动匹配引擎

#### 4.3.1 需求描述

系统自动将导入的实收记录与登记的应收款进行匹配，减少用户手工核对的工作量。

#### 4.3.2 功能要点

| 编号 | 功能点 | 描述 |
|------|--------|------|
| F-3.1 | 精确匹配 | 基于"金额完全一致 + 交易时间在预期到账日期前后 3 天内"进行精确匹配 |
| F-3.2 | 模糊匹配 | 专业版支持基于备注、客户名称的模糊匹配（如备注包含客户名拼音缩写） |
| F-3.3 | 多对一匹配 | 支持多笔实收合并匹配一笔应收（如客户分两次付款） |
| F-3.4 | 一对多匹配 | 支持一笔实收拆分匹配多笔应收（如客户一次性支付多个项目的费用） |
| F-3.5 | 匹配置信度 | 为每条匹配结果标注置信度（高/中/低），帮助用户优先确认高置信度匹配 |
| F-3.6 | 人工修正 | 用户可手动修正匹配结果（取消匹配、重新匹配），修正结果反馈给匹配引擎优化后续匹配 |
| F-3.7 | 自动匹配触发 | 每次导入新数据后自动触发匹配，也可手动触发重新匹配 |

#### 4.3.3 匹配规则

1. **金额匹配**：实收金额与应收金额完全一致，或差异在允许范围内（如手续费扣除）
2. **时间匹配**：实收时间在预期到账日期前后 N 天内（N 可配置，默认 3 天）
3. **备注匹配**（专业版）：实收备注包含应收款编号、客户名称、项目名称等关键字
4. **历史模式**：基于该客户历史交易的金额、时间规律，辅助匹配

#### 4.3.4 验收标准

- 精确匹配准确率 ≥ 95%
- 模糊匹配准确率 ≥ 85%（专业版）
- 匹配引擎处理速度：1000 条数据 ≤ 5 秒

### 4.4 差异标记与分析

#### 4.4.1 需求描述

系统自动识别并标记应收与实收之间的差异，帮助用户快速定位问题。

#### 4.4.2 差异类型分类

| 差异类型 | 定义 | 系统标记方式 |
|----------|------|--------------|
| 未到账 | 应收款超过预期到账日期仍无对应实收 | 红色标记 + "未到账"标签 |
| 超额到账 | 实收金额大于应收金额 | 黄色标记 + "超额"标签 |
| 手续费扣除 | 实收金额小于应收金额，差异符合平台手续费规则 | 蓝色标记 + "手续费"标签 |
| 汇率差异 | 多币种交易中，因汇率波动导致的差异 | 橙色标记 + "汇率差"标签 |
| 金额不符 | 实收金额与应收金额存在差异，且不属于上述类型 | 紫色标记 + "金额不符"标签 |
| 不明差异 | 系统无法自动分类的差异 | 灰色标记 + "待确认"标签 |

#### 4.4.3 功能要点

| 编号 | 功能点 | 描述 |
|------|--------|------|
| F-4.1 | 差异自动标记 | 系统根据差异类型自动标记，并在对账看板中按颜色区分 |
| F-4.2 | 差异详情 | 点击差异条目可查看详细信息：应收金额、实收金额、差异金额、差异类型、关联的应收/实收记录 |
| F-4.3 | 差异处理 | 用户可对差异条目进行处理：标记为"已解决""待跟进""忽略"，并添加处理备注 |
| F-4.4 | 差异统计 | 按差异类型、客户、项目、平台维度统计差异金额和笔数 |
| F-4.5 | 差异追溯 | 差异条目可追溯到原始应收记录和实收记录，便于核实 |

#### 4.4.4 验收标准

- 差异自动分类准确率 ≥ 90%
- 差异标记实时展示，无延迟
- 支持按差异类型筛选和排序

### 4.5 月度对账报告

#### 4.5.1 需求描述

系统按月生成对账报告，汇总本月应收、实收、差异情况，支持多维度查看和导出。

#### 4.5.2 功能要点

| 编号 | 功能点 | 描述 |
|------|--------|------|
| F-5.1 | 报告自动生成 | 每月 1 号自动生成上月对账报告，用户也可手动生成指定月份的报告 |
| F-5.2 | 多维度汇总 | 支持按客户、项目、平台、差异类型 多维度汇总 |
| F-5.3 | 报告内容 | 报告包含：本月应收总额、实收总额、差异总额、差异明细、未到账清单、手续费汇总 |
| F-5.4 | 导出格式 | 支持 PDF（用于存档和发送给客户）和 Excel（用于进一步分析）两种格式导出 |
| F-5.5 | 报告模板 | 专业版支持自定义报告模板（如添加公司 Logo、调整报告结构） |
| F-5.6 | 历史报告 | 支持查看历史所有已生成的报告，便于纵向对比 |

#### 4.5.3 验收标准

- 报告生成时间 ≤ 10 秒（1000 条数据以内）
- PDF 导出排版正确，无乱码
- 历史报告保留期限：免费版 3 个月，专业版永久

### 4.6 报税导出

#### 4.6.1 需求描述

为自由职业者提供报税所需的收入汇总数据和开票清单，简化报税准备流程。

#### 4.6.2 功能要点

| 编号 | 功能点 | 描述 |
|------|--------|------|
| F-6.1 | 收入汇总表 | 按年度/季度汇总收入数据，包括：总收入、各平台收入、各币种收入、手续费扣除汇总 |
| F-6.2 | 开票清单 | 汇总已开票和未开票记录，支持按客户、项目筛选 |
| F-6.3 | 报税格式 | 导出格式符合国家税务总局要求的个人所得税申报格式 |
| F-6.4 | 多币种汇总 | 专业版支持将多币种收入按年度平均汇率折算为人民币汇总 |
| F-6.5 | 导出格式 | 支持 Excel 和 PDF 两种格式导出 |

#### 4.6.3 验收标准

- 导出的报税数据可直接用于个人所得税申报
- 多币种折算准确率 ≥ 99%（基于年度平均汇率）
- 仅专业版用户可使用报税导出功能

## 6. 业务规则

### 6.1 匹配规则

1. 精确匹配的优先级高于模糊匹配
2. 一笔实收记录只能匹配一笔应收记录（一对多匹配场景除外）
3. 匹配结果需用户确认后才正式生效（高置信度匹配可配置为自动确认）
4. 用户手动修正的匹配结果优先级最高

### 6.2 差异判定规则

1. 差异金额 = |应收金额 - 实收金额|
2. 手续费差异的判定阈值：差异金额 ≤ 应收金额 × 平台标准手续费率（可配置）
3. 汇率差异的判定阈值：差异金额 ≤ 应收金额 × 1%（可配置）
4. 超出阈值的差异标记为"金额不符"，需用户人工核实

### 6.3 版本限制规则

1. 免费版每月最多导入 50 笔流水，超出后无法继续导入（次月 1 号重置）
2. 免费版最多绑定 2 个收款渠道
3. 免费版不支持多币种、报税导出功能
4. 免费版历史数据保留 3 个月，超出部分无法查看（但不会删除）

## 7. 数据需求

### 7.1 CSV 数据格式

#### 7.1.1 通用字段

| 字段名 | 是否必填 | 说明 |
|--------|----------|------|
| 交易时间 | 是 | 格式：YYYY-MM-DD HH:MM:SS 或 YYYY-MM-DD |
| 交易金额 | 是 | 正数为收入，负数为支出 |
| 交易对象 | 否 | 付款方名称/账号 |
| 交易备注 | 否 | 附言、备注信息 |
| 交易状态 | 否 | 成功/失败/处理中 |
| 交易 ID | 否 | 平台流水号，用于去重 |
| 币种 | 否 | 默认 CNY，多币种场景必填 |

#### 7.1.2 平台特有字段

- **微信**：交易类型、商户订单号、支付方式
- **支付宝**：业务类型、商户订单号、账户余额
- **Stripe**：Payment Intent ID、Card Brand、Country
- **PayPal**：Transaction ID、Transaction Status、Currency Code

### 7.2 数据存储

| 数据类型 | 存储期限 | 说明 |
|----------|----------|------|
| 免费版用户数据 | 3 个月 | 超出部分无法查看但不删除 |
| 专业版用户数据 | 永久 | 用户注销后 30 天内彻底删除 |
| 导入日志 | 6 个月 | 用于追溯导入问题 |
| 操作日志 | 1 年 | 用于安全审计 |

## 8. 约束与假设

### 8.1 约束

1. **MVP 开发周期**：约 7 天，需优先实现核心功能（CSV 导入 + 匹配引擎 + 对账看板 + PDF 导出）
2. **技术栈限制**：前端采用 Web 技术栈，后端采用轻量级框架，确保快速迭代
3. **合规约束**：不得存储用户的平台登录凭证，只能通过 CSV 导入方式获取数据

### 8.2 假设

1. 用户能够自行从各收款平台导出 CSV 账单文件
2. 各平台的 CSV 格式相对稳定，不会频繁变更
3. 用户具备一定的财务基础知识，能够理解应收/实收/差异等概念
4. MVP 阶段以个人用户为主，暂不考虑团队协作场景
