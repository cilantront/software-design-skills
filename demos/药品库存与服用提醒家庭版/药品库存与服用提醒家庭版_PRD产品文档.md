# 1 概述

## 1.1 需求背景

随着中国老龄化进程加速，家庭药箱管理已成为一个普遍而刚性的痛点。据调查，超过60%的家庭存在药品过期未发现、漏服忘服、药品库存混乱等问题。对于需要长期用药的慢病患者（高血压、糖尿病等）和负责照护的子女而言，"记不住吃什么药、不知道药还有没有、不清楚药是否过期"是日常生活中反复出现的困扰。

现有市场上的药品管理工具大多面向医院或药房场景，功能复杂、学习成本高，缺乏针对家庭场景的轻量化工具。微信小程序作为国民级应用，覆盖率高、使用门槛低，是承载家庭药品管理功能的理想载体。

本产品旨在打造一款**小众但刚需**的家庭药品管理工具，核心价值主张为：**不做医疗诊断，只做家庭药品库存管理和服药提醒**。通过扫码/拍照快速录入、智能服药提醒、库存告警与补药清单、家庭照护协同四大核心能力，帮助家庭用户实现药品管理的数字化和规范化。

**预期达成目标：**
1. MVP阶段5-7天完成开发上线
2. 免费版支持1名成员+20种药品，降低用户尝试门槛
3. 家庭版（¥12/月）解锁多成员协同和云同步，实现商业变现
4. 核心操作路径（录入药品、确认服药）不超过3步，适合老年人使用

## 1.2 名词解释

| **名词** | **说明** |
| --- | --- |
| 用药成员 | 需要长期服药的家庭成员，是服药提醒的直接接收者 |
| 照护者 | 负责远程照护的家庭成员（如子女），可查看患者服药情况并接收漏服通知 |
| 家庭管理员 | 创建家庭账号、邀请成员加入的高级角色，通常由家庭中负责统筹的子女担任 |
| OCR | Optical Character Recognition，光学字符识别，用于从药品包装照片中提取文字信息 |
| 服药提醒窗口 | 从提醒时间开始，到判定为漏服的时间范围（默认30分钟，可配置） |
| 临期阈值 | 用户自定义的天数，当药品有效期距当前日期小于等于该值时触发临期提醒（默认30天） |
| 最低库存线 | 用户可按药品设置的剩余数量下限，当实际数量低于该值时触发库存不足提醒 |
| 补药清单 | 系统自动汇总的需补充药品列表，包含已触发临期或库存不足提醒的药品 |

| 漏服 | 在设定的服药提醒窗口内，用药成员未确认服药的状态 |
| 补服 | 漏服后，由照护者手动标记为"已补服"的操作 |

## 1.3 产品介绍

### 1.3.1 范围说明

| 项 | 内容 |
| --- | --- |
| 包含功能 | 药品录入（扫码/拍照OCR/手动）、药品列表与详情管理、服药提醒设置与通知、服药确认与漏服检测、服药记录与统计、库存告警（临期/不足/紧急）、补药清单生成与管理、家庭账号创建与成员管理、数据共享与同步、订阅管理（免费版/家庭版）、数据导出、提醒偏好设置 |
| 不包含功能 | 医疗诊断功能、用药建议或处方功能、在线问诊、药品购买（电商功能）、医院/药房系统对接、药品不良反应报告 |

本产品面向以下目标用户：
- **需要长期用药的老人家庭及其子女**：父母需要每天服用多种慢病药物，子女希望远程了解父母服药情况
- **多成员家庭中负责照护的人**：需要协调多位家人的用药管理
- **慢病患者（高血压、糖尿病等）**：需要规律服药并追踪服药记录，以便就医时提供给医生参考

产品核心价值：
1. **降低录入门槛**：扫码/拍照即可识别药品信息，老年人无需手动输入
2. **服药不遗漏**：定时提醒+漏服通知+家属确认，形成完整的服药闭环
3. **药品不浪费**：实时监控有效期和库存，临期预警避免药品过期
4. **家庭共照护**：子女可远程查看父母服药情况，实现跨地域的家庭关怀

# 3 产品功能

## 3.1 用户端小程序功能

### 3.1.1 药品扫码录入

功能描述：用户通过扫描药品包装上的条形码，系统自动匹配药品库信息（药品名称、规格、厂家），匹配成功后自动填充，用户仅需补充有效期、数量和服用说明。

优先级与依赖说明：

| 项 | 内容 |
| --- | --- |
| 优先级 | P0 |
| 依赖需求 | URS-3.1.1 扫码录入 |
| 前置条件 | 用户已登录小程序，已授权摄像头权限，已选择用药成员 |

### 3.1.2 药品扫码录入—详细流程

```mermaid
flowchart TD
    A[用户点击扫码录入] --> B[调起摄像头扫码界面]
    B --> C[用户扫描药品条形码]
    C --> D[系统向药品条码数据库发起查询]
    D --> E{3秒内是否匹配到?}
    E -->|是| F[自动填充药品名称/规格/厂家]
    E -->|否| G[提示未匹配到，引导手动录入]
    F --> H[用户确认或修改药品信息]
    G --> H
    H --> I[用户补充有效期/数量/服用说明]
    I --> J[保存到药品库存]
    J --> K{是否设置服药提醒?}
    K -->|是| L[跳转提醒设置]
    K -->|否| M[录入完成]
```

业务规则说明：
1. 扫码响应时间不超过3秒，超时自动切换为手动录入模式
2. 条码匹配结果仅供参考，用户始终可修改所有字段
3. 同一药品（相同条码或相同名称+规格）不可重复录入，提示已存在并引导编辑
4. 有效期必须晚于当前日期，否则提示"药品已过期，是否仍要录入？"

### 3.1.3 药品扫码录入—主要原型

[药品扫码录入原型](assets/prototypes/drug-scan-input-widget.html)

验收标准说明：
- [ ] 正常流程：扫描已知条码后3秒内自动填充药品信息，用户可修改并保存
- [ ] 正常流程：扫描未知条码后提示手动录入，手动填写后可正常保存
- [ ] 异常流程：网络超时3秒后自动切换手动录入模式
- [ ] 异常流程：药品已存在时弹出提示并提供编辑入口
- [ ] 性能要求：扫码到展示结果不超过3秒

### 3.1.4 药品拍照OCR录入

功能描述：用户拍摄药品包装或说明书照片，系统通过OCR识别药品名称、有效期、规格等关键信息，展示识别结果供用户确认或修改。

优先级与依赖说明：

| 项 | 内容 |
| --- | --- |
| 优先级 | P0 |
| 依赖需求 | URS-3.1.1 拍照OCR录入 |
| 前置条件 | 用户已登录小程序，已授权摄像头权限，已选择用药成员 |

### 3.1.5 药品拍照OCR录入—详细流程

```mermaid
flowchart TD
    A[用户点击拍照录入] --> B[调起相机拍照界面]
    B --> C[用户拍摄药品包装/说明书]
    C --> D[系统调用OCR识别服务]
    D --> E[5秒内返回识别结果]
    E --> F{识别是否成功?}
    F -->|是| G[展示识别出的药品名称/有效期/规格]
    F -->|否| H[提示识别失败，引导手动录入或重拍]
    G --> I[用户确认或修改识别结果]
    H --> I
    I --> J[用户补充数量/服用说明]
    J --> K[保存到药品库存]
```

业务规则说明：
1. OCR识别结果必须展示给用户确认，不允许直接写入正式库存
2. 识别响应时间不超过5秒
3. 用户可逐项修改识别结果中的每个字段
4. 支持重新拍照替换当前识别结果

### 3.1.8 手动录入药品—详细流程

