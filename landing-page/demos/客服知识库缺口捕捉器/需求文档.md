## 2. 产品概述

### 2.1 产品定位

> **只做一件事：从真实客服对话中发现知识库缺口，并帮你补齐。**

KB-Gap Catcher 是一款轻量级客服运营外挂工具。它不替代 Zendesk、Intercom、飞书客服台等平台，而是读取这些平台上已有的对话/工单数据，通过 LLM 聚类与语义分析，自动识别出"知识库没覆盖、客服在反复口头解释、用户反复被转人工"的问题，并输出一份可直接执行的"知识库补全任务清单"。

### 2.2 产品愿景

让小团队的客服知识库像开源项目一样"从真实使用中持续生长"，而非等到问题爆发后才被动修补。

### 2.3 核心价值主张

| # | 价值 | 说明 |
|---|------|------|
| 1 | **从被动到主动** | 不再等用户投诉或客服反馈才补文档，而是系统性地从每一次对话中挖掘缺失 |
| 2 | **从经验到数据** | 用频次 + 影响用户数说话，而非凭"感觉哪里不对" |
| 3 | **从发现到行动** | 不只给报告，直接生成 FAQ 草稿和待审核任务，缩短闭环 |
| 4 | **从重型到轻量** | 不替换现有客服系统，5 分钟接入，即插即用 |

### 2.4 产品边界（做什么 / 不做什么）

| ✅ 做 | ❌ 不做 |
|-------|---------|
| 对话/工单的批量导入与分析 | 完整的客服工作台 / 工单系统 |
| 知识库缺口的自动识别与排序 | 直接面向终端用户的聊天机器人 |
| FAQ / 标准回复草稿生成 | 知识库的终端用户侧搜索/展示 |
| 审核协作与任务看板 | 客服排班、绩效、质检 |
| 多渠道数据的接入适配器 | 深度替换 Zendesk / Intercom 等平台 |

## 3. 目标用户分析

### 3.1 用户画像

#### 画像 A：小团队客服运营（主要）

| 维度 | 描述 |
|------|------|
| **典型角色** | 运营专员 / 客服组长 / 联合创始人兼客服 |
| **团队规模** | 5-50 人 SaaS / 电商 / 在线教育团队 |
| **日常职责** | 维护 FAQ、帮助中心、客服话术库；培训新客服；处理升级投诉 |
| **工具栈** | 飞书客服台 / 企微客服 / Intercom / Zendesk / 自建工单系统 |
| **痛点烈度** | ★★★★★ — 知识库维护是"重要但永远排不上优先级"的事 |
| **技术水平** | 非技术背景，能导入 CSV、使用 SaaS 工具，不写代码 |

#### 画像 B：客服主管 / 产品经理（次要）

| 维度 | 描述 |
|------|------|
| **典型角色** | 客服主管、产品经理、客户成功经理 |
| **关注点** | 客户满意度、首次解决率、客服效率 |
| **使用方式** | 查看缺口报告、审批 FAQ 草稿、推动知识库迭代 |

#### 画像 C：一线客服（间接受益）

| 维度 | 描述 |
|------|------|
| **典型角色** | 一线客服坐席 |
| **间接收益** | 知识库更完整 → 更少"答不上来"的尴尬 → 更少转人工 → 工作压力降低 |

### 3.2 用户特征总结

- 团队规模小，没有专职知识库运营岗
- 客服系统已有，但知识库是"Excel/飞书文档/Notion 表格"级别的半成品
- 对"AI 能力"有好感但无技术能力自建
- 预算敏感，期望工具价格 < 一个实习生日薪

## 4. 用户场景与痛点

### 4.1 场景 1：周末大促后的知识库废墟

> **角色**：某电商团队客服运营小王（团队 12 人）
>
> **场景**：618 大促结束后，小王面对 3000+ 条客服对话。大促期间上新了 20 多个 SKU，新增了退换货政策变更，但知识库还是大促前的版本。她需要逐条翻看对话，找出"客服答不上来、反复@主管"的问题，手动整理成 FAQ 补充到飞书文档。
>
> **痛点**：
> - 逐条翻对话耗时 2-3 天，每次大促后都是"还债周"
> - 容易遗漏——真正高频的问题可能被淹没在噪音中
> - 整理出来的 FAQ 质量参差不齐，缺少统一格式

### 4.2 场景 2：反复被问的"隐藏高频问题"

> **角色**：某 SaaS 公司客户成功经理 Lisa（团队 8 人）
>
> **场景**：每周都有用户问"能不能导出 PDF 报告"，客服都是口头回复"暂不支持，但你可以用截图"。这个问题从没被写进 FAQ，因为没人意识到它在每周出现 30+ 次。
>
> **痛点**：
> - 客服已经形成"口头答案"，但从未沉淀到知识库
> - 新客服不知道这个"口头答案"，每次都要问老客服
> - 产品团队看不到这类"隐性需求信号"

### 4.3 场景 3：知识库有，但没人信

> **角色**：某在线教育平台客服组长阿杰（团队 25 人）
>
> **场景**：帮助中心有 200+ 篇 FAQ，但客服普遍反映"搜不到"或"搜到的不对"。阿杰不知道哪些 FAQ 已经过时、哪些高频问题根本就没写。
>
> **痛点**：
> - 无法量化"知识库哪里不行"——只能凭感觉
> - 缺少"对话证据"来说服产品/业务团队更新内容
> - 知识库的维护变成"谁喊得响谁优先"

### 4.4 场景 4：新系统上线后的知识冷启动

> **角色**：某跨境电商客服负责人（团队 30 人，刚迁移到 Intercom）
>
> **场景**：从旧的邮件工单系统迁移到 Intercom，原有的 FAQ 是散落在 Word 文档里的。需要快速建立一套新的知识库，但不知道从哪里开始。
>
> **痛点**：
> - 历史对话分散在旧系统，需要批量导出再分析
> - 冷启动时不知道"哪些问题是高频的"
> - 时间紧迫，需要在 1-2 周内完成知识库初版

### 4.5 痛点总结矩阵

| 痛点 | 影响面 | 频次 | 当前替代方案 |
|------|--------|------|-------------|
| 手动翻对话找缺口 | ★★★★★ | 每周/每次大促后 | Excel + 人工 + 飞书文档 |
| 高频问题未被沉淀 | ★★★★ | 持续发生 | 口头传承、老带新 |
| 知识库过时但无法量化 | ★★★★ | 持续发生 | 年度大扫除（往往也不做） |
| 缺少对话证据推动更新 | ★★★ | 持续发生 | 截图、凭经验争论 |
| 知识库冷启动无方向 | ★★★ | 系统迁移/新产品上线 | 拍脑袋 + 加班 |

## 5. 用户需求清单

### US-01：批量导入对话数据

> 作为 **客服运营**，我希望 **能批量导入客服对话和工单数据（CSV/文本文件）**，以便 **不用逐条翻看就能分析所有对话**。

**验收标准：**
- Given 用户上传一个 CSV 文件（含对话内容列）
- When 系统解析完成
- Then 显示解析统计（总条数、成功条数、失败条数）并提供失败行预览

### US-02：自动识别知识库缺口

> 作为 **客服运营**，我希望 **系统能自动识别对话中"客服答不上来/反复转人工/临时口头解释"的问题**，以便 **我不需要逐条阅读就能发现知识库缺口**。

**验收标准：**
- Given 已导入一批对话
- When 分析完成
- Then 系统输出一份"缺口清单"，每条缺口包含：问题摘要、出现频次、影响用户数、代表性对话证据（≥2 条）

### US-03：按优先级排序缺口

> 作为 **客服运营**，我希望 **缺口清单按"频次 × 影响用户数"排序**，以便 **优先解决影响最大的问题**。

**验收标准：**
- Given 已生成缺口清单
- When 用户查看清单
- Then 默认按综合优先级排序，且可切换为按频次、影响用户数、时间等维度排序

### US-04：一键生成 FAQ 草稿

> 作为 **客服运营**，我希望 **能一键为缺口生成 FAQ 草稿和标准回复**，以便 **节省从零撰写的时间**。

**验收标准：**
- Given 用户选中一条或多条缺口
- When 点击"生成 FAQ 草稿"
- Then 系统基于对话上下文生成 FAQ 草稿（含问题标题、答案正文、标签），支持人工编辑后确认

### US-05：审核协作与任务看板

> 作为 **客服主管**，我希望 **能在任务看板上分配、审核、追踪 FAQ 的编写与发布**，以便 **知识库更新有流程、有责任人**。

**验收标准：**
- Given 已生成 FAQ 草稿
- When 主管将其分配到看板
- Then 草稿进入"待审核 → 审核通过/打回 → 已发布"流程，每一步有时间戳和操作人

### US-06：知识库历史版本

> 作为 **客服主管**，我希望 **能看到知识库条目的历史版本**，以便 **在需要时回溯或回滚**。

**验收标准：**
- Given 一条 FAQ 被多次编辑
- When 用户查看该 FAQ 的历史
- Then 展示完整版本列表，支持版本对比和回滚

### US-07：多渠道数据导入

> 作为 **客服运营**，我希望 **能从飞书、企微、Intercom、Zendesk 等渠道导入对话**，以便 **覆盖团队使用的所有客服渠道**。

**验收标准：**
- Given 用户配置了某渠道的 API 凭据或上传了该渠道的导出文件
- When 触发同步
- Then 系统成功拉取/解析对话数据，并标记数据来源渠道

| F-02.5 | 综合优先级评分 | P0 | 频次 × 影响用户数 × 时间衰减，生成优先级排序 |

## 7. 非功能需求

### 7.1 性能要求

| 指标 | 要求 |
|------|------|
| 单次导入上限 | 免费版 500 条对话，团队版 10,000 条 |
| 分析耗时 | 500 条对话 ≤ 5 分钟完成分析 |
| 页面加载 | 首屏 ≤ 2 秒（国内网络环境） |
| 并发支持 | 支持 50 个团队同时在线分析 |

### 7.2 安全与隐私

| 要求 | 说明 |
|------|------|
| 数据隔离 | 多租户架构，团队间数据严格隔离 |
| 传输加密 | 全链路 HTTPS/TLS 1.2+ |
| 存储加密 | 对话数据 AES-256 加密存储 |
| 数据删除 | 用户可随时删除数据，30 天内彻底清除 |
| LLM 数据安全 | 对话数据发给 LLM 分析时，脱敏处理用户 PII（姓名、手机号、地址等） |
| 权限控制 | 基于角色的访问控制（RBAC） |

### 7.3 可用性

| 要求 | 说明 |
|------|------|
| 可用性 SLA | 99.5%（团队版） |
| 数据备份 | 每日自动备份，保留 30 天 |
| 移动端 | MVP 阶段仅 Web 端，后续考虑移动端查看 |

### 7.4 兼容性

| 维度 | 要求 |
|------|------|
| 浏览器 | Chrome 90+, Firefox 88+, Safari 14+, Edge 90+ |
| 分辨率 | 最小支持 1280×720，推荐 1920×1080 |
| 国际化 | MVP 仅中文，v1.1 支持英文界面 |

### 7.5 可观测性

| 要求 | 说明 |
|------|------|
| 分析日志 | 每次分析任务记录输入、输出、耗时、LLM token 消耗 |
| 错误告警 | 分析失败时自动告警并保留原始数据 |
| 用量统计 | 管理员可查看本月对话分析量、FAQ 生成量 |

## 11. 竞争分析与差异化定位

### 11.1 竞品定位矩阵

```
                    重量级 / 完整客服系统
                         ▲
                         │
          Zendesk ●      │      ● Intercom
                         │
                         │
          传统客服 ●      │      ● 网易七鱼
                         │
                         │
    ◄────────────────────┼────────────────────►
    通用工具              │           垂直/外挂工具
                         │
                         │
              KB-Gap     │
             Catcher ●   │
                         │
          自建脚本 ●     │
                         │
                         ▼
                    轻量级 / 单点工具
```

### 11.2 差异化优势

| 维度 | KB-Gap Catcher | Zendesk/Intercom | 自建脚本 |
|------|---------------|-----------------|---------|
| **定位** | 知识库缺口发现的垂直工具 | 完整客服平台 | 临时方案 |
| **接入成本** | 5 分钟（上传 CSV 即可） | 数月实施 | 需开发资源 |
| **AI 能力** | 开箱即用的 LLM 聚类+FAQ 生成 | 附加 AI 模块（贵） | 需自建 prompt |
| **价格** | ¥0-79/月 | ¥数百-数千/月 | 人力成本 |
| **替换成本** | 零（外挂工具） | 替换现有系统 | 维护负担 |
| **专注度** | 只做知识库缺口 | 什么都做 | 只做一次 |

### 11.3 竞争壁垒

1. **专注定位**：不做大而全的客服系统，只做"知识库缺口发现"这一个点，做到极致
2. **数据飞轮**：积累的"对话→缺口→FAQ"数据可持续优化聚类与生成质量
3. **渠道生态**：与飞书/企微/Intercom/Zendesk 形成互补而非竞争，更易获得生态支持
4. **价格护城河**：¥79/月的定价让大平台难以跟进（他们的 AI 模块通常按坐席收费）

## 12. 验收标准

### 12.1 MVP 验收标准

| 编号 | 验收项 | Given | When | Then |
|------|--------|-------|------|------|
| AC-01 | CSV 导入 | 用户上传符合模板的 CSV（≤500 条） | 点击导入 | 系统解析成功并显示统计（成功/失败条数） |
| AC-02 | 缺口识别 | 已导入含"客服答不上来"场景的对话 | 分析完成 | 系统输出缺口清单，每条含摘要、频次、影响用户数、≥2 条证据 |
| AC-03 | 语义聚类 | 分析完成 | 查看缺口清单 | 相似问题（如"怎么退款"和"钱什么时候退"）被归为同一缺口 |
| AC-04 | 优先级排序 | 缺口清单已生成 | 查看清单 | 默认按"频次×影响用户数"降序排列 |
| AC-05 | FAQ 草稿生成 | 选中一条缺口 | 点击"生成 FAQ 草稿" | 系统生成 FAQ 草稿（含标题、答案、标签），可在编辑器中修改 |
| AC-06 | 任务看板 | FAQ 草稿已生成 | 拖拽到"进行中"列 | 状态更新成功，其他视图同步 |
| AC-07 | 免费额度 | 免费版用户 | 月度用量达 500 条 | 系统提示额度已满，引导升级 |

| NFR-01 | 分析性能 | 500 条对话分析 ≤ 5 分钟 |
| NFR-02 | 数据安全 | 对话数据加密存储，LLM 调用时 PII 脱敏 |
| NFR-03 | 浏览器兼容 | Chrome/Firefox/Safari/Edge 最新两个大版本正常显示与操作 |
| NFR-04 | 响应式 | 1280px 以上宽度正常使用 |

## 13. 风险与假设

### 13.1 假设

| # | 假设 | 验证方式 |
|---|------|---------|
| H1 | 小团队确实有"知识库更新不及时"的痛点 | MVP 用户访谈 + 落地页注册转化 |
| H2 | 用户愿意为"自动生成 FAQ 草稿"付费 | 定价页 A/B 测试 |
| H3 | LLM 生成的 FAQ 草稿质量足以节省人工时间 | 用户满意度调查（草稿采纳率 > 60% 为达标） |
| H4 | CSV 导入是 MVP 阶段的足够入口 | 首月导入方式统计 |

### 13.2 风险

| # | 风险 | 影响 | 缓解措施 |
|---|------|------|---------|
| R1 | LLM API 成本过高 | 免费版亏损 | 设置免费额度上限，使用缓存减少重复分析 |
| R2 | LLM 聚类质量不稳定 | 用户体验差 | 引入人工反馈循环，持续优化 prompt |
| R3 | 渠道 API 变更 | 同步功能失效 | MVP 不做渠道 API，先验证 CSV 路径 |
| R4 | 数据隐私顾虑 | 用户不敢上传对话 | PII 脱敏、数据加密、明确隐私政策 |
| R5 | 大平台跟进 | 竞争压力 | 保持专注和轻量，做平台"不屑于做"的小事 |

## 14. 术语表

| 术语 | 定义 |
|------|------|
| **知识库缺口** | 客服在实际对话中遇到的、现有知识库未覆盖或覆盖不足的问题 |
| **FAQ 草稿** | 系统基于对话上下文自动生成的 FAQ 初稿，需人工审核后发布 |
| **语义聚类** | 将语义相似的问题归为同一类别的技术（如"怎么退款"和"钱什么时候退"） |
| **对话证据** | 支持某个缺口存在的原始对话片段 |
| **影响用户数** | 在分析周期内，遇到同一缺口的不同用户数量 |
| **缺口优先级** | 综合频次、影响用户数、时间衰减等因素计算的排序分值 |
| **PII** | Personally Identifiable Information，个人可识别信息（姓名、手机号、地址等） |
| **LLM** | Large Language Model，大语言模型 |
